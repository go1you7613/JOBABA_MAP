# JOBABA MAP 개발팀 전달 소스 범위

작성일: 2026-07-20

## 1. 적용 원칙

잡아바 운영 원천 채용 데이터와 적재 배치는 개발팀의 현행 운영 구조를 그대로 사용합니다.

- 운영 원천 데이터 적재 방식은 기존 `INSERT`/`UPDATE` 방식을 유지합니다.
- MAP 소스는 운영 원천 테이블을 생성하거나 초기화하지 않습니다.
- MAP 소스는 원천 API를 직접 수집하거나 CSV를 원천 테이블에 적재하지 않습니다.
- MAP 소스는 운영 원천 테이블에 `DROP`, `TRUNCATE`, `DELETE`, `RENAME`을 실행하지 않습니다.
- staging/old/prev 테이블 교체와 별도 롤백 테이블을 사용하지 않습니다.

## 2. 개발팀 전달 대상

### 애플리케이션

- 사용자 지도 화면과 정적 리소스
- 지도 REST API
- 지도 조회 Service/DAO/MyBatis Mapper
- 지도 검색·응답 VO

### MAP 신규 DB 객체

- `tb_empmn_map_coord`: 채용공고 좌표 캐시
- `tb_jobcls_ncs_map`: NCS와 잡아바 직종코드 매핑

### 원천 조회 계약

- `v_job_posting`: 채널별 운영 원천 테이블을 MAP 공통 컬럼으로 읽는 조회 객체
- `db/v_job_posting_prod_template.sql`: 운영 원천을 변경하지 않는 VIEW 정의 예시

조회 계약 템플릿에는 공공데이터포털 원천인 `jwrki.tb_public_job` 채널이 포함되어야 합니다.

## 3. 전달 대상에서 제거한 항목

- 고용24 OpenAPI 원천 수집기
- 고용24 공공기관 별도 수집기 및 `tb_work24_public_job`
- Worknet CSV 적재기
- 문서/CSV 기반 로컬 원천 적재기
- `v_job_posting_staging`, `v_job_posting_old`, `tb_empmn_worknet_api_prev`
- `v_job_posting` 전체 재생성·교체 실행 스크립트
- 운영 원천 테이블을 로컬 형태로 재생성하는 DDL

## 4. 개발팀 확인 후 확정할 항목

다음 항목은 실제 잡아바 개발서버 소스와 DB 스키마를 받은 뒤 확정합니다.

1. 운영 원천 테이블별 실제 PK와 공고 식별자
2. `USE_YN`, `DEL_YN` 등 논리 상태 컬럼
3. 고용24 `INFO_SVC`의 공공/민간 코드 정의
4. 공공데이터포털 `jwrki.tb_public_job`의 실제 컬럼과 사용 조건
5. 공공데이터포털 공고의 주소 및 좌표 연결 기준
6. 채널별 직종·경력·학력·고용형태 코드 변환 기준
7. 운영에 이미 존재하는 통합 채용 조회 객체의 재사용 가능 여부

## 5. 적용 순서

1. 개발팀 제공 소스와 로컬 잡아바 개발환경을 동기화합니다.
2. 실제 운영 원천 스키마를 기준으로 조회 계약을 보정합니다.
3. MAP 신규 객체와 애플리케이션 소스만 잡아바 모듈에 반영합니다.
4. 로컬 잡아바 개발서버에서 공공·민간 채널과 주요 화면을 검증합니다.
5. 검증 결과를 개발팀에 전달한 뒤 잡아바 개발서버 배포 여부를 별도로 결정합니다.

## 6. 배포 제한

이 변경은 GitHub 소스 전달까지만 수행합니다.

- 개인 EC2 서버에 배포하지 않습니다.
- 잡아바 개발서버에도 자동 배포하지 않습니다.
- 잡아바 개발서버 배포는 개발팀 소스 동기화와 로컬 검증이 끝난 후 별도 승인으로 진행합니다.
