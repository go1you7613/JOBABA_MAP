# Python Secure Coding Guide 2023

이 문서는 국내 `Python 시큐어코딩 가이드(2023년 개정본)`를 Python 개발 시 적용 가능한 실전 점검 기준으로 정리한 것이다. 구현 중에는 `docs/SECURITY/SECURITY_VIBE_CODING_GUIDE.md`와 `docs/SECURITY/SECURITY_VIBE_CODING_GUIDE_PYTHON.md`를 우선 적용하고, 판단이 애매하거나 보안 민감 변경이면 이 문서를 상세 기준으로 참조한다.

## 적용 원칙

- [ ] 보안 요구사항은 기능 요구사항과 같은 완료 조건으로 적용한다.
- [ ] 사용자 입력, 외부 API 응답, 파일, 환경변수, 메시지 큐, DB 저장값을 기본적으로 신뢰하지 않는다.
- [ ] “방어가 있다”, “고려한다”, “가능하면” 같은 표현으로 완료 처리하지 않는다.
- [ ] 위험 기능을 사용해야 하면 사용 이유, 제한 조건, 보완 통제를 코드 리뷰에 남긴다.

## 1. 입력 데이터 검증 및 표현

### SQL 삽입

- [ ] SQL, JPQL 유사 DSL, raw query에는 파라미터 바인딩을 적용한다.
- [ ] 사용자 입력을 SQL 문자열에 직접 결합하지 않는다.
- [ ] 동적 컬럼명, 정렬 방향, 테이블명은 서버 측 허용목록으로 매핑한다.

### 코드 삽입

- [ ] `eval()`, `exec()`, `compile()`에 사용자 입력을 전달하지 않는다.
- [ ] 동적 import, plugin loading, expression evaluation은 허용목록과 서명 검증 또는 관리자 승인 절차를 적용한다.
- [ ] 템플릿 표현식, filter, macro를 사용자 입력으로 구성하지 않는다.

### 명령 삽입

- [ ] `subprocess`, `os.system`, `popen`에 사용자 입력을 직접 전달하지 않는다.
- [ ] `subprocess.run([...], shell=False)` 형태를 사용한다.
- [ ] 실행 가능한 명령과 옵션은 서버 측 허용목록으로 제한한다.

### 경로 조작

- [ ] 파일 경로는 base directory를 정하고 canonical path가 base directory 안인지 검증한다.
- [ ] `../`, 절대 경로, URL 인코딩 우회, 심볼릭 링크 우회를 검증한다.
- [ ] 사용자 제공 파일명은 저장 파일명으로 직접 사용하지 않는다.

### 크로스사이트 스크립팅

- [ ] 템플릿 자동 이스케이프를 비활성화하지 않는다.
- [ ] `safe`, `Markup`, `|safe`는 sanitizer와 허용목록이 적용된 값에만 사용한다.
- [ ] 사용자 HTML을 허용해야 하면 허용 태그, 속성, 프로토콜을 명시적으로 제한한다.

### 위험한 형식 문자열

- [ ] 로그, SQL, 명령어, 템플릿, 경로 생성에 사용자 입력을 형식 문자열로 직접 삽입하지 않는다.
- [ ] 로그에는 구조화 필드를 사용하고 비밀값과 개인정보를 마스킹한다.

### HTTP 응답 분할

- [ ] 헤더 값에는 CR, LF 문자를 허용하지 않는다.
- [ ] redirect URL과 다운로드 파일명은 허용목록 또는 안전 인코딩을 적용한다.

### 부적절한 인코딩 및 이스케이프

- [ ] HTML, JavaScript, URL, SQL, Shell, CSV 컨텍스트별로 맞는 인코딩을 적용한다.
- [ ] 하나의 escape 함수로 모든 컨텍스트를 처리하지 않는다.

## 2. 보안 기능

### 부적절한 인증

- [ ] 보호 대상 API, View, Route에는 인증 장치를 반드시 적용한다.
- [ ] 인증 우회용 debug endpoint, test account, 임시 token은 운영에 포함하지 않는다.
- [ ] 로그인 실패, 계정 잠금, 비밀번호 재설정은 enumeration이 발생하지 않게 응답을 통일한다.

### 부적절한 인가

- [ ] 객체 조회, 수정, 삭제에는 서버 측 소유권 또는 역할 검사를 수행한다.
- [ ] 클라이언트가 전달한 user_id, role, tenant_id를 권한 판단 근거로 신뢰하지 않는다.
- [ ] 관리자 기능은 URL 숨김이 아니라 서버 권한 검사로 보호한다.

### 중요한 자원에 대한 잘못된 권한 설정

- [ ] 업로드, 로그, 백업, 임시 파일은 웹 루트 바깥에 저장한다.
- [ ] 생성 파일과 디렉터리는 최소 권한으로 생성한다.
- [ ] secret, private key, DB 파일은 저장소와 정적 리소스 경로에 두지 않는다.

### 취약한 암호화 알고리즘

- [ ] MD5, SHA1, DES, 3DES, RC4를 보안 목적에 사용하지 않는다.
- [ ] 비밀번호는 bcrypt, argon2id, PBKDF2 같은 password hashing 알고리즘으로 저장한다.
- [ ] 암호화 키는 코드에 하드코딩하지 않고 secret 저장소 또는 환경변수로 주입한다.

### 중요 정보 평문 저장 및 전송

- [ ] 비밀번호, 토큰, 주민등록번호, 계좌번호, API key를 평문 저장하지 않는다.
- [ ] HTTPS를 적용하고 운영 쿠키에는 Secure, HttpOnly, SameSite를 설정한다.
- [ ] 로그와 오류 응답에 secret, token, 개인정보를 출력하지 않는다.

### 하드코딩된 비밀값

- [ ] DB 계정, API key, JWT secret, encryption key를 코드와 테스트 fixture에 하드코딩하지 않는다.
- [ ] 샘플 값은 실제 형식과 혼동되지 않도록 명확한 dummy 값으로 작성한다.

### 취약한 난수

- [ ] 토큰, 임시 비밀번호, nonce, 세션 식별자에는 `secrets` 또는 OS CSPRNG를 사용한다.
- [ ] `random` 모듈은 보안 목적에 사용하지 않는다.

## 3. 시간 및 상태

### 경쟁 조건

- [ ] 잔액, 포인트, 재고, 쿠폰, 권한 변경은 트랜잭션과 잠금 또는 원자적 update로 처리한다.
- [ ] 중복 요청 방지를 위해 idempotency key, unique constraint, 상태 전이 검사를 적용한다.

### 세션 고정

- [ ] 로그인 성공 후 세션 ID를 재발급한다.
- [ ] 권한 상승 또는 관리자 전환 후 세션 상태를 갱신한다.

### CSRF

- [ ] 상태 변경 요청은 GET으로 구현하지 않는다.
- [ ] 쿠키 기반 인증의 POST, PUT, PATCH, DELETE 요청에는 CSRF 토큰 검증을 적용한다.
- [ ] AJAX 상태 변경 요청에도 CSRF 토큰 헤더 또는 동등한 서버 검증을 적용한다.
- [ ] SameSite 쿠키 정책만으로 CSRF 방어를 완료 처리하지 않는다.

## 4. 에러 처리

- [ ] 사용자에게 stacktrace, SQL 오류, 내부 경로, 환경변수, secret을 노출하지 않는다.
- [ ] 운영 응답은 일반화된 오류 메시지를 사용하고 상세 원인은 서버 로그에만 남긴다.
- [ ] 예외를 삼켜서 보안 실패를 성공처럼 처리하지 않는다.
- [ ] 인증, 권한, 결제, 파일 처리 실패 로그에는 추적 가능한 식별자를 남긴다.

## 5. 코드 오류

### Null 및 타입 오류

- [ ] Optional, None, 빈 문자열, 빈 리스트 상태를 명확히 처리한다.
- [ ] 타입 변환 실패가 권한 우회나 검증 누락으로 이어지지 않게 기본값을 신중히 설정한다.

### 자원 누수

- [ ] 파일, socket, DB cursor, connection은 context manager로 닫는다.
- [ ] 외부 API와 DB 호출에는 timeout을 설정한다.

### 부적절한 초기화

- [ ] 보안 설정 기본값은 안전한 값으로 초기화한다.
- [ ] 설정 누락 시 debug 또는 allow-all로 fallback하지 않는다.

## 6. 캡슐화 및 API 오용

- [ ] 내부 객체, secret, permission flag를 응답 DTO에 그대로 노출하지 않는다.
- [ ] mutable 전역 상태에 사용자별 인증 정보를 저장하지 않는다.
- [ ] framework 보안 기능을 끄는 decorator, middleware 예외, 설정 변경은 리뷰 없이 추가하지 않는다.

## 7. 역직렬화 및 파일 파서

- [ ] 신뢰할 수 없는 데이터에는 `pickle`, `marshal`, `shelve`를 사용하지 않는다.
- [ ] YAML은 `yaml.safe_load()`를 사용한다.
- [ ] XML은 XXE 방어 라이브러리 또는 안전 설정을 적용한다.
- [ ] 압축 파일 해제 시 대상 경로 검증과 총 크기 제한을 적용한다.

## 8. 의존성 점검

- [ ] 새 패키지 추가 전 공식 저장소, 유지보수 상태, 라이선스, 최근 CVE를 확인한다.
- [ ] lock 파일을 유지하고 버전을 무제한 범위로 열어두지 않는다.
- [ ] `pip-audit`, `safety`, `poetry audit` 중 프로젝트 표준 도구를 실행한다.

## 9. Python 개발 완료 체크리스트

- [ ] 인증 대상 endpoint가 익명 접근으로 열려 있지 않다.
- [ ] 권한 대상 리소스가 IDOR에 취약하지 않다.
- [ ] 입력값 검증이 서버에서 수행된다.
- [ ] SQL, 명령어, 경로, 템플릿에 사용자 입력을 직접 결합하지 않는다.
- [ ] CSRF, CORS, 세션, 쿠키 설정을 변경한 경우 보안 기준을 검증했다.
- [ ] 파일 업로드, 다운로드, 압축 해제, 파서 처리에 크기와 경로 제한을 적용했다.
- [ ] debug, stacktrace, secret, 개인정보 노출을 제거했다.
- [ ] 의존성 취약점 점검을 실행했거나 실행하지 못한 사유를 남겼다.
