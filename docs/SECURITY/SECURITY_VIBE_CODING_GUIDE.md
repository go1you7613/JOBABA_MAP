# Security Vibe Coding Guide

이 문서는 AI와 함께 기능을 설계, 구현, 수정할 때 매번 보안 요구사항을 같이 검증하기 위한 범용 가이드다. 특정 언어, 프레임워크, 클라우드, 데이터베이스에 종속되지 않는다.

## 사용 방법

이 파일은 프로젝트의 `docs/SECURITY/` 폴더에 두고, `AGENTS.md`, `CLAUDE.md`, `CONTRIBUTING.md` 중 에이전트가 매번 읽는 문서에서 링크한다.

권장 문구:

```markdown
보안 관련 개발, 수정, 리뷰, QA 작업을 할 때는 `docs/SECURITY/SECURITY_VIBE_CODING_GUIDE.md`를 기준으로 보안 영향 분류, 체크리스트, 완료 전 보안 게이트를 적용한다.
```

언어별 프로젝트에서는 다음 확장 문서를 함께 적용한다.

- Java, Spring, JSP, Thymeleaf, Maven, Gradle 프로젝트: `docs/SECURITY/SECURITY_VIBE_CODING_GUIDE_JAVA.md`
- Python, Django, Flask, FastAPI, Jinja2, SQLAlchemy 프로젝트: `docs/SECURITY/SECURITY_VIBE_CODING_GUIDE_PYTHON.md`

## 공통 적용성 판정

이 문서의 보안 원칙, 웹취약점 기준, 개인정보 기준, 공급망 기준, 완료 전 보안 게이트는 Java와 Python에 공통 적용한다.

다만 다음 항목은 언어와 프레임워크별 구현 방식이 다르므로 공통 파일만으로 완료 처리하지 않는다. 해당 언어별 확장 문서의 지침까지 반드시 적용한다.

- 인증과 인가 미들웨어
- CSRF 토큰 처리
- 입력값 검증 방식
- SQL 파라미터 바인딩 방식
- 템플릿 이스케이프와 XSS 방어
- 파일 업로드 처리 API
- XML, YAML, JSON 역직렬화 API
- 의존성 취약점 점검 명령
- 테스트와 QA 검증 방법

## 목적

- 기능 개발 중 보안 검토를 나중으로 미루지 않는다.
- 웹취약점, 소스코드 보안약점, 개인정보 보호, 운영 설정을 같은 흐름에서 확인한다.
- AI가 코드를 작성할 때 보안 관련 가정, 위험, 검증 결과를 명시하게 한다.
- 결과물은 "동작한다"에서 끝나지 않고 "안전하게 동작한다"까지 확인한다.

## 적용 범위

이 문서는 다음 작업에 적용한다.

- 신규 기능 개발
- 기존 기능 수정
- 버그 수정
- API 추가 또는 변경
- 인증, 권한, 세션, 파일, 결제, 관리자 기능 변경
- DB 스키마, 쿼리, 배치, 마이그레이션 변경
- 외부 API, 오픈소스, SDK, AI API 연동
- 배포 설정, 환경변수, 로그, 모니터링 변경

## 기본 원칙

1. 보안은 별도 단계가 아니라 구현 조건이다.
2. 요구사항이 불명확하면 보안에 불리한 방향으로 추정하지 않는다.
3. 사용자 입력은 신뢰하지 않는다.
4. 클라이언트 검증은 편의 기능이고, 서버 검증이 기준이다.
5. 인증과 인가는 별개다.
6. 권한검사는 화면이 아니라 서버에서 한다.
7. 비밀값은 코드, 로그, 문서, 테스트 데이터에 남기지 않는다.
8. 개인정보는 최소 수집, 최소 보관, 최소 노출을 기본값으로 한다.
9. 에러는 사용자에게 친절하게, 내부에는 충분히 기록하되 민감정보는 남기지 않는다.
10. 완료 보고에는 보안 검증 결과를 포함한다.

## AI 에이전트 행동 규칙

AI는 개발 전에 다음을 먼저 확인한다.

- 이 변경이 인증, 권한, 개인정보, 파일, 외부 연동, 결제, 관리자 기능에 영향을 주는가
- 새 입력값, 새 URL, 새 API, 새 DB 쿼리, 새 파일 처리, 새 로그가 생기는가
- 기존 보안 패턴, 미들웨어, 헬퍼, 공통 유틸이 있는가
- 프로젝트의 보안 관련 문서가 있는가

AI는 불확실한 경우 다음을 명시한다.

- Assumption: 현재 가정
- Risk: 틀렸을 때 생기는 위험
- Decision needed: 사용자 확인이 필요한 지점

AI는 보안 항목을 `검토함`, `고려함`, `문제없음` 같은 표현만으로 완료 처리하지 않는다. 반드시 적용한 통제, 적용하지 못한 사유, 대체 통제, 검증 방법 중 하나를 구체적으로 남긴다.

AI는 다음 상황에서 구현 전에 질문한다.

- 권한 정책이 모호한 경우
- 개인정보 수집 또는 표시 범위가 모호한 경우
- 외부 API 키, 토큰, 인증정보 처리 방식이 모호한 경우
- 파일 업로드 허용 범위가 모호한 경우
- 운영 보안 설정을 낮추는 변경이 필요한 경우
- 보안 검증을 생략해야 하는 경우

## 개발 흐름

### 1. 요구사항 분석

작업 시작 시 다음을 정리한다.

- 변경 목적
- 영향받는 사용자
- 영향받는 데이터
- 새로 생기는 입력 경로
- 새로 생기는 출력 경로
- 필요한 권한 정책
- 성공 기준
- 보안 검증 기준

### 2. 보안 영향 분류

다음 중 하나라도 해당하면 보안 민감 변경으로 본다.

- 로그인, 로그아웃, 회원, 관리자, 세션, 토큰
- 권한, 역할, 소유자 검증
- 개인정보, 위치정보, 결제정보, 신용정보
- 파일 업로드, 다운로드, 미리보기, 이미지 처리
- 검색, 필터, 정렬, 동적 쿼리
- 외부 API, 웹훅, OAuth, SSO
- 관리자 화면, 운영자 기능, 대량 다운로드
- 환경변수, 배포 설정, CORS, 보안 헤더
- 로그, 감사기록, 모니터링
- 오픈소스 패키지 추가 또는 버전 변경

보안 민감 변경이면 최소한 다음을 수행한다.

- 권한 정책 확인
- 입력값 검증 확인
- 에러와 로그 확인
- 테스트 또는 수동 검증 추가
- 완료 보고에 보안 검증 결과 포함

### 3. 구현

구현 중에는 다음 순서를 따른다.

1. 기존 보안 패턴을 찾는다.
2. 기존 공통 유틸을 우선 사용한다.
3. 입력값 검증을 먼저 설계한다.
4. 권한검사를 비즈니스 로직 가까이에 둔다.
5. DB 접근은 파라미터 바인딩을 사용한다.
6. 출력은 컨텍스트에 맞게 이스케이프한다.
7. 로그에는 식별자 위주로 남기고 민감정보는 제외한다.
8. 테스트 또는 재현 가능한 검증 방법을 남긴다.

### 4. 검증

완료 전 다음을 확인한다.

- 기능 정상 동작
- 인증 없는 접근 차단
- 권한 없는 접근 차단
- 잘못된 입력값 처리
- 민감정보 노출 없음
- 콘솔/서버 로그에 민감정보 없음
- 새 의존성 취약점 여부 확인
- 배포 설정에 보안 약화 없음

## 웹취약점 점검표

이하 모든 점검표의 체크 항목은 권장사항이 아니라 기본 준수 항목이다. 코드리뷰, QA, 보안 점검 중 체크 항목을 만족하지 않는 구현을 발견하면 보안 결함으로 지적하고 수정한다. 예외가 필요한 경우에는 예외 사유, 남는 위험, 대체 통제를 작업 기록에 남긴다.

### 인증

- [ ] 보호 대상 URL/API에는 인증을 반드시 적용한다.
- [ ] 로그인 실패 응답은 계정 존재 여부를 구분할 수 없도록 동일한 메시지와 동일한 상태 코드로 처리한다.
- [ ] 비밀번호는 Argon2, bcrypt, PBKDF2 등 검증된 비밀번호 해시 알고리즘으로 저장한다.
- [ ] 비밀번호 재설정 토큰은 보안용 난수 생성기로 만들고 만료 시간을 반드시 설정한다.
- [ ] 로그아웃은 서버 세션 또는 토큰 상태를 무효화한다.

### 인가

- [ ] 관리자 기능은 관리자 권한을 서버에서 검증한다.
- [ ] 리소스 ID로 조회할 때 소유자 또는 접근 권한을 서버에서 반드시 검증한다.
- [ ] 화면에서 버튼을 숨기는 것만으로 권한을 처리하지 않는다.
- [ ] 대량 조회, export, 통계 API도 권한검사를 한다.
- [ ] URL 직접 접근으로 타인 데이터에 접근하는 요청은 서버 권한검사로 차단한다.

### 세션과 쿠키

- [ ] 세션 쿠키에 `HttpOnly`를 적용한다.
- [ ] HTTPS 환경에서는 `Secure`를 적용한다.
- [ ] `SameSite` 정책을 명시한다.
- [ ] 세션 만료 시간을 반드시 설정한다.
- [ ] 권한 변경 후 기존 세션 무효화 또는 재인증 정책을 반드시 구현한다.

### 입력값 검증

- [ ] 서버에서 필수값, 길이, 형식, 범위를 검증한다.
- [ ] 숫자 ID는 숫자로 변환한 뒤 사용한다.
- [ ] enum, status, role, sort 값은 allowlist로 검증한다.
- [ ] 검색어와 필터값은 파라미터 바인딩으로 처리하고, 정렬 컬럼과 방향은 allowlist로만 반영한다.
- [ ] HTML, URL, 파일명, 경로 입력은 별도 검증한다.

### Injection

- [ ] SQL은 파라미터 바인딩 또는 안전한 ORM API를 사용한다.
- [ ] 동적 컬럼명, 테이블명, 정렬 방향은 allowlist만 허용한다.
- [ ] OS 명령 실행에 사용자 입력을 직접 전달하지 않는다.
- [ ] 템플릿, XPath, LDAP, NoSQL 쿼리에 사용자 입력을 직접 삽입하지 않는다.
- [ ] CSV/Excel export에 사용자 입력이 포함되면 `=`, `+`, `-`, `@`로 시작하는 값을 반드시 중화한다.

### XSS

- [ ] 사용자 입력 출력 시 템플릿/프레임워크의 자동 이스케이프를 해제하지 않는다.
- [ ] HTML을 허용해야 하면 sanitize 정책을 사용한다.
- [ ] `innerHTML`, `dangerouslySetInnerHTML`, `|safe`는 금지한다. 불가피하면 sanitize 적용과 허용 사유를 코드리뷰에 남긴다.
- [ ] HTML 속성, URL, JavaScript 문자열에 값을 삽입할 때는 컨텍스트별 인코딩을 반드시 적용한다.
- [ ] 사용자 생성 콘텐츠를 표시하는 화면에는 Content Security Policy를 적용한다. 적용하지 못하면 사유와 대체 통제를 기록한다.

### CSRF

- [ ] 데이터 생성, 수정, 삭제, 승인, 권한 변경, 로그아웃 등 상태 변경 요청은 GET으로 구현하지 않는다. 반드시 POST, PUT, PATCH, DELETE 중 하나를 사용한다.
- [ ] 쿠키 기반 인증을 쓰는 모든 상태 변경 요청에는 CSRF 토큰 검증을 반드시 적용한다.
- [ ] AJAX/fetch/XHR로 보내는 상태 변경 요청에도 CSRF 토큰을 헤더 또는 요청 본문에 포함하고, 서버에서 반드시 검증한다.
- [ ] CSRF 토큰 방식을 적용하기 어려운 API는 Origin 또는 Referer 검증, 재인증, 커스텀 헤더 검증 등 동등한 서버 측 방어를 반드시 적용한다.
- [ ] `SameSite` 쿠키 정책은 보조 방어로만 사용한다. `SameSite`만으로 CSRF 방어를 대체하지 않는다.
- [ ] QA 또는 코드리뷰에서 CSRF 토큰 검증이 없는 쿠키 기반 상태 변경 요청을 발견하면 보안 취약점으로 지적하고 수정한다.

### 파일 업로드와 다운로드

- [ ] 파일 업로드는 서버에서 허용 확장자 allowlist를 반드시 검증한다.
- [ ] 업로드 파일의 MIME 타입과 파일 시그니처를 서버에서 반드시 검증한다.
- [ ] 파일 크기 제한을 반드시 적용한다.
- [ ] 저장 파일명은 사용자 파일명을 그대로 쓰지 않는다.
- [ ] 업로드 파일은 실행 경로 밖에 저장한다.
- [ ] 다운로드 전 권한검사를 한다.
- [ ] 경로를 요청 파라미터로 직접 받지 않는다.

### API

- [ ] 객체 수준 권한검사를 한다.
- [ ] API 응답은 allowlist DTO 또는 명시 필드 매핑으로 구성하고, 내부 객체 전체를 그대로 반환하지 않는다.
- [ ] 로그인, 인증, 관리자, 대량 조회, export, 고비용 API에는 Rate limit 또는 abuse 방어를 적용한다.
- [ ] CORS는 필요한 origin만 허용한다.
- [ ] 에러 응답에 스택트레이스, SQL, 내부 경로를 노출하지 않는다.

### 보안 헤더

- [ ] `Content-Security-Policy`를 적용한다. 적용하지 못하면 사유와 대체 통제를 기록한다.
- [ ] `X-Content-Type-Options: nosniff`를 적용한다.
- [ ] `X-Frame-Options` 또는 CSP `frame-ancestors`를 적용한다.
- [ ] `Referrer-Policy`를 적용한다.
- [ ] HTTPS 운영 환경에서는 `Strict-Transport-Security`를 적용한다.
- [ ] 불필요한 서버, 프레임워크 버전 헤더를 노출하지 않는다.

## 소스코드 보안약점 점검표

### 입력데이터 검증 및 표현

- [ ] SQL에 사용자 입력을 문자열 결합, 템플릿 문자열, format 함수로 직접 삽입하지 않는다.
- [ ] 사용자 입력을 HTML로 출력할 때 이스케이프 또는 sanitize를 반드시 적용한다.
- [ ] 파일 경로 입력은 정규화 후 허용된 기준 디렉터리 내부인지 반드시 검증한다.
- [ ] OS 명령 실행이 필요하면 사용자 입력을 명령 문자열에 직접 결합하지 않고 안전한 인자 배열과 allowlist를 사용한다.
- [ ] 파일 업로드 제한은 확장자, MIME, 파일 시그니처, 크기를 서버에서 모두 검증한다.
- [ ] 파일 다운로드는 파일 ID로만 처리하고, 서버에서 경로 조회와 권한검사를 수행한다.
- [ ] 서버가 사용자 입력 URL로 외부 요청을 보내는 기능은 allowlist, 사설 IP 차단, 리다이렉트 제한, timeout을 반드시 적용한다.
- [ ] 응답 헤더에 사용자 입력을 넣을 때 CR/LF 문자를 제거하거나 거부한다.

### 보안기능

- [ ] 보호 대상 엔드포인트, 화면, API에는 서버 측 인증 미들웨어, 필터, 데코레이터 중 프로젝트 표준 인증 장치를 반드시 적용한다.
- [ ] 관리자 기능, 소유자 리소스, 상태 변경 기능에는 서버 측 권한검사를 반드시 적용한다.
- [ ] 비밀번호, 토큰, API 키를 평문 저장하지 않는다.
- [ ] MD5, SHA-1, DES, ECB 모드처럼 안전하지 않은 암호 알고리즘과 모드는 사용하지 않는다.
- [ ] 랜덤값은 보안용 난수 생성기를 사용한다.
- [ ] 비밀값을 코드에 하드코딩하지 않는다.
- [ ] 민감정보는 전송 중 TLS로 보호한다.

### 시간 및 상태

- [ ] 중복 요청, 재전송, race condition이 금전, 권한, 상태값에 영향을 주는 로직에는 트랜잭션, 잠금, 멱등키 중 필요한 방어를 적용한다.
- [ ] 결제, 포인트, 재고, 권한 변경은 원자성을 보장한다.
- [ ] 토큰과 인증코드는 만료 시간과 1회성 사용을 반드시 적용한다.
- [ ] 반복문, 배치, 외부 호출, 검색 API에는 timeout, 개수 제한, 페이지네이션 중 필요한 자원 제한을 적용한다.

### 에러 처리

- [ ] 사용자에게 내부 예외 메시지를 그대로 반환하지 않는다.
- [ ] 예외 처리로 인증/인가 실패가 정상 처리처럼 흐르지 않는다.
- [ ] 실패 시 안전한 기본값을 사용한다.
- [ ] 로그에는 원인 추적에 필요한 식별자와 상황만 남긴다.

### 코드 오류

- [ ] null, undefined, empty 상태를 처리한다.
- [ ] 타입 변환 실패를 처리한다.
- [ ] 자원은 사용 후 해제한다.
- [ ] 외부 호출에는 timeout과 실패 처리를 반드시 적용한다.
- [ ] 인증, 인가, 결제, 개인정보, 파일 처리 분기에는 테스트 또는 재현 가능한 수동 검증 절차를 남긴다.

### 캡슐화와 정보노출

- [ ] 디버그 코드, 테스트 계정, 임시 우회 코드를 제거한다.
- [ ] 내부 경로, 설정, 환경변수, 스택트레이스를 노출하지 않는다.
- [ ] 객체 전체를 응답으로 반환하지 않는다.
- [ ] 로그, 이벤트, 분석 도구로 민감정보가 전송되지 않는다.

### API 오용

- [ ] deprecated 또는 취약한 API를 사용하지 않는다.
- [ ] 인증, 암호화, 파싱, 직렬화는 검증된 라이브러리를 사용한다.
- [ ] 역직렬화에 신뢰할 수 없는 입력을 그대로 넣지 않는다.
- [ ] XML 파싱 시 XXE 방어를 적용한다.

## 개인정보 및 민감정보 점검표

### 분류

변경 대상 데이터가 다음에 해당하는지 확인한다.

- 개인정보
- 고유식별정보
- 민감정보
- 위치정보
- 신용정보
- 인증정보
- 결제정보
- 기업 또는 기관의 비공개 정보

### 처리 원칙

- [ ] 개인정보 수집 목적을 기능 요구사항 또는 정책 문서에 명시한다.
- [ ] 수집 목적 수행에 필요한 필드만 수집하고, 불필요한 필드는 제거한다.
- [ ] 보관 기간과 파기 기준을 반드시 정의한다.
- [ ] 화면 표시 시 마스킹한다.
- [ ] 다운로드와 export에는 최소 필드만 포함한다.
- [ ] 로그에 개인정보를 남기지 않는다.
- [ ] 테스트 데이터는 실제 개인정보를 사용하지 않는다.
- [ ] 운영 데이터 접근 권한을 제한한다.

### 암호화와 키 관리

- [ ] 비밀번호는 복호화 불가능한 해시로 저장한다.
- [ ] 주민번호, 계좌번호, 인증 토큰, API 키, 복구 코드는 저장 전 암호화하거나 해시 처리한다.
- [ ] 암호화 키는 코드와 분리한다.
- [ ] 암호화 키의 교체, 폐기, 접근권한 정책을 반드시 정의한다.
- [ ] 백업 데이터도 보호한다.

## 오픈소스 및 공급망 점검표

- [ ] 새 패키지를 추가하기 전에 표준 라이브러리나 기존 의존성으로 대체할 수 없는지 확인하고, 추가 사유를 기록한다.
- [ ] 공식 패키지명과 저장소가 일치하지 않거나 typosquatting이 의심되면 패키지를 추가하지 않는다.
- [ ] 최근 유지보수 상태가 불량한 패키지는 대체 패키지를 사용한다. 불가피하면 사유와 보완책을 기록한다.
- [ ] 알려진 High/Critical 취약점이 있는 패키지는 추가하지 않는다. 불가피하면 예외 승인과 완화책을 기록한다.
- [ ] 라이선스가 프로젝트 정책과 맞지 않는 패키지는 추가하지 않는다.
- [ ] lockfile 변경에서 의도하지 않은 패키지 추가, major 버전 변경, registry 변경이 있으면 제거하거나 사유를 기록한다.
- [ ] 의존성 목록 또는 SBOM을 생성할 수 있는 명령과 산출물 위치를 기록한다.
- [ ] 패키지 추가 이유를 PR 또는 작업 기록에 남겼다.

## 운영 설정 점검표

- [ ] 운영 환경에서는 debug, devtools, detailed error 옵션을 반드시 비활성화한다.
- [ ] 비밀값은 환경변수 또는 시크릿 매니저로 관리한다.
- [ ] CORS는 필요한 origin만 허용한다.
- [ ] 관리자 페이지에는 인증, 관리자 권한검사, 접근 경로 제한 중 필요한 보호조치를 적용한다.
- [ ] 헬스체크는 내부 정보를 노출하지 않는다.
- [ ] 에러 페이지는 내부 정보를 노출하지 않는다.
- [ ] 로그 보관 기간과 접근 권한을 반드시 정의한다.
- [ ] 백업, 복구, 장애 대응 절차를 반드시 정의하고 운영 문서에 남긴다.

## AI 연동 기능 점검표

AI 기능이 포함되면 다음을 추가로 확인한다.

- [ ] 프롬프트에 개인정보나 비밀값을 넣지 않는다.
- [ ] 사용자 입력을 시스템 지시문처럼 신뢰하지 않는다.
- [ ] 도구 호출 권한을 최소화한다.
- [ ] 모델 응답을 그대로 명령, SQL, HTML로 실행하지 않는다.
- [ ] 출력값 검증과 후처리를 적용한다.
- [ ] 대화 로그 저장 범위와 보관 기간을 정한다.
- [ ] 외부 AI API로 개인정보를 전송하려면 법적 근거와 사용자 동의를 확인하고, 근거가 없으면 전송하지 않는다.
- [ ] prompt injection과 data exfiltration 방어를 적용한다. 최소한 시스템 지시문 보호, 도구 호출 allowlist, 외부 전송 데이터 제한을 구현한다.

## 작업 유형별 필수 보안 질문

### 새 API를 만들 때

- 누가 호출할 수 있는가
- 어떤 권한이 필요한가
- 어떤 입력값을 받는가
- 어떤 데이터가 응답에 포함되는가
- Rate limit 또는 abuse 방어를 적용할 대상인가
- 에러 응답에 내부 정보가 없는가

### DB 쿼리를 추가할 때

- 사용자 입력이 쿼리에 들어가는가
- 파라미터 바인딩을 사용하는가
- 동적 정렬, 필터, 컬럼명은 allowlist인가
- 대량 조회 제한이 있는가
- 개인정보 필드 접근이 필요한가

### 화면을 수정할 때

- 사용자 입력이 출력되는가
- HTML 렌더링을 허용하는가
- 마스킹이 필요한 데이터가 있는가
- 숨긴 버튼 외에 서버 권한검사가 있는가
- 브라우저 콘솔에 민감정보가 찍히지 않는가

### 파일 기능을 만들 때

- 허용 파일 형식은 무엇인가
- 최대 크기는 얼마인가
- 저장 위치는 안전한가
- 다운로드 권한검사는 어디서 하는가
- 원본 파일명을 화면이나 응답 헤더에 표시할 때 인코딩과 CR/LF 제거를 적용하는가

### 관리자 기능을 만들 때

- 관리자 역할이 세분화되어 있는가
- 생성, 수정, 삭제, 승인, 권한 변경 작업에 감사 로그를 남기는가
- 대량 변경 또는 삭제의 확인 절차가 있는가
- 권한 변경 이력이 남는가
- 관리자 기능 API가 일반 사용자에게 노출되지 않는가

## 완료 전 보안 게이트

AI는 작업 완료 전에 다음 형식으로 자체 점검한다.

```text
Security gate:
- Auth/authz impact: none | checked | changed
- Input validation: none | checked | changed
- Sensitive data: none | checked | changed
- Injection/XSS/CSRF: checked
- File/API/external call risk: none | checked | changed
- Logs/secrets: checked
- Dependency risk: none | checked
- Verification: <tests, commands, browser checks, or manual reasoning>
```

`changed`가 하나라도 있으면 어떤 파일에서 어떤 방어를 적용했는지 설명한다.

## 완료 보고 형식

```text

보안 반영:
- <적용한 보안 조치>

검증:
- <실행한 테스트 또는 확인 방법>

남은 위험:
- 없음
- 또는 <남은 위험과 이유>
```

## 보안 검증을 생략할 수 있는 경우

다음처럼 보안 영향이 없는 변경은 간단히 기록하고 넘어갈 수 있다.

- 오탈자 수정
- 순수 문서 수정
- 스타일만 바꾸는 CSS 변경
- 테스트 설명 문구 수정

단, 다음은 작은 변경이어도 보안 검토를 생략하지 않는다.

- URL, API, 폼, 쿼리, 파일 처리 변경
- 인증, 권한, 세션 변경
- 개인정보 표시 또는 다운로드 변경
- 환경변수, 로그, 배포 설정 변경
